## 0. 一句话定位

**Apolla AI 是一款面向个人知识工作的 AI 工作台：自动研究、自动整理、自动生成成品，并在你授权下执行低风险操作。**

它不是"又一个多模型聊天壳"，也不是一开始就做全自治"AI 员工"。它的核心价值闭环是 **可信研究（Trusted Research）→ 一键成品（Instant Artifact）→ 低风险执行（Low-risk Action）**。

---

## 1. 背景与机会

当前 To C AI 产品竞争呈三种范式：

| 产品 | 本质 | 赢在 | 主要短板 |
|---|---|---|---|
| Monica | 低摩擦分发型多模型助手 | **入口**（浏览器侧边栏、快捷键） | 产品线散、团队能力弱、偏"助手"而非"长任务执行" |
| Genspark | Workspace 型 AI 员工 | **工作台**（Slides/Docs/Sheets、MCP 生态、团队治理） | 产品面太宽、学习成本高、credits 心智复杂 |
| Manus | 任务执行型行动引擎 | **执行引擎**（沙箱、Browser Operator、Projects/Skills） | 信任门槛高、验证码/确认摩擦、成本感知不直观 |

**机会点**：三者各有最重的负担。Apolla 的策略是**组合三者长处、避开三者重负**——入口学 Monica，工作台与模板学 Genspark，异步任务/项目/低风险自动化学 Manus；但商业上避免 Genspark 式"超级宽产品"，避免 Manus 式"高自治先行"的安全与信任成本。

**核心产品决策**：
- **MVP 先做"研究与交付"闭环，V1 再做"执行与自动化"闭环。**
- 先把"搜得准、写得好、出得快、可复用、可分享、成本可控"打穿，再向浏览器自动化、计划任务、个人记忆、Custom Skills 扩张。

## 2. 目标用户与场景

首发聚焦**高频知识工作者与轻度创作/自动化用户**，不做泛大众。

| 用户画像 | 主要 JTBD | 高频场景 | 购买动机 |
|---|---|---|---|
| 内容创作者/营销 | 快速调研、写稿、出图、出 PPT、竞品拆解 | 热点研究、内容策划、品牌方案、渠道文案、竞品周报 | 节省研究与制作时间，提升产出密度 |
| 学生/研究员/求职者 | 阅读长文、整理资料、产出结构化结果 | 论文综述、课程报告、面试准备、简历/作品集、学习卡片 | 把碎片信息变成可提交成果 |
| 独立开发者/PM | 竞品分析、PRD 原型、网页生成、需求拆解 | 需求写作、产品原型、技术调研、可视化脚本、Demo 页面 | 更快从想法到原型与材料 |
| 轻自动化用户 | 把重复网页/文件工作自动化 | 表单填写、网页抽取、批量整理、个人日报/周报 | 减少机械劳动，保留人工决策 |

**北极星指标**：**每个留存用户每周完成的"有效工作流"数**（比 MAU 或消息数更能反映产品是否进入用户工作链路）。

## 3. 产品价值主张

用户买的不是 GPT 入口，也不是 Claude 入口，而是三个结果：

1. **更可信**：带来源、能回溯、能解释。"来源透明 + 观点对比 + 结构化证据面板"是默认体验，而非附加项。
2. **更成品化**：结果默认输出成"可发给别人"的形态（Doc/PDF/PPT/网页 Artifact/表格/任务清单），而不是结束于一段聊天。
3. **更可执行**：但执行 = **低风险自动化 + 高风险确认制**，不是无边界自治。

## 4. 三条核心用户流（首版只重点打磨这三条）

### 用户流一：研究到成品（北极星流程，最高优先级）
```
输入问题 / 上传材料
  → 系统拆解问题（生成研究计划）
  → 搜索与抓取（带来源）
  → 生成证据视图（来源列表 + 引用片段 + 观点对比）
  → 用户点选"生成报告 / Slides / 网页"
  → 交付成品
  → 分享 / 导出
```

### 用户流二：当前页增强（增长入口，承接 Monica 的高频优势）
```
网页中打开侧边栏
  → 总结 / 翻译 / 抽取 / 比对
  → 保存到项目
  → 继续对话
  → 生成 Artifact 或写入文档
```

### 用户流三：模板化复用（留存核心，承接 Genspark Workflows / Manus Skills）
```
保存某次高质量任务为模板 / Skill
  → 下次在新资料上复跑
  → 自动生成新结果
  → 按需改写
  → 导出
```

## 6. 功能详细需求（P0 — MVP 范围）

### 6.1 统一聊天 + 模型路由
- 模式切换：`Auto`（默认，系统智能路由）/ `GPT` / `Claude`。
- 所有普通聊天**流式输出**，p95 首字延迟 ≤ 3s（GA ≤ 2s）。
- 内部采用**逻辑别名**而非硬编码模型名（详见架构文档），用户侧只暴露 Auto/GPT/Claude 三种心智。
- 默认路由策略：

| 场景 | 默认策略 |
|---|---|
| 普通聊天/轻总结 | `gpt_fast` |
| 长文改写/邮件/文案/语气润色 | `claude_write` |
| Deep Research 规划 / 最终综合 | 规划 `gpt_premium`，成稿/润色 `claude_write` |
| 结构化抽取/表格/API 输出 | `gpt_premium` + Structured Outputs |
| 多工具/多步复杂代理 | `claude_premium` 或 `gpt_premium`（按评测择优）|
| 失败降级 | 回退到轻量模型 + 缩上下文 + 只读模式 |

### 6.2 Web Research（可信研究）
- 输入问题 → 自动拆解为子问题 → 搜索 → 抓取 → 抽取 → 比对 → 综合。
- **每条结论必须可回溯到来源**：来源列表、引用片段、观点对比面板。
- 长任务走**异步任务对象**：前端先显示"计划草图 + 阶段进度 + 预估耗时"，后台补齐结果。
- 验收：来源引用覆盖率 ≥ 80%（MVP）/ ≥ 95%（GA）。

### 6.3 文件理解
- 支持 PDF / Word / 网页 / 图片上传，多轮问答。
- 文件解析 + OCR + 切分 + 嵌入，结果进入项目素材库可复用。

### 6.4 Projects（项目空间）
- 每个项目有独立上下文、素材库、历史任务。
- 上下文继承：项目内新任务自动带入项目背景。

### 6.5 Artifact 生成
- 生成可运行网页 / 交互 demo / 小工具。
- 可预览、可编辑、可分享、可发布为公开网页。

### 6.6 Docs / Slides 导出
- 一键导出 Doc / PDF / PPT / 网页。
- 导出件保留来源引用。

### 6.7 浏览器侧边栏（MVP 范围严格受限）
- **首版只做：浏览器侧边栏 + 页面 DOM 理解 + 低风险辅助动作**（总结/翻译/抽取/比对/保存到项目/生成报告）。
- **首版不做**：云浏览器登录控制、替用户持有会话（信任与客服成本过高，留待 v1+）。

### 6.8 计费与配额
- Freemium / Pro / Power 三档。
- **任务级成本提示**：高成本任务（Deep Research、长文、Artifact）显式标识。
- 每次用户任务都是可观察、可计费、可回放、可归档的"任务对象"。

### 6.9 反馈与评测
- 点赞 / 点踩 / 纠错 / 结果追踪，数据回流用于模型路由优化与 LLM eval golden set。

## 7. 自动化权限分级（安全核心）

动作分三级，**MVP 只开放前两级，且低风险写入必须显式确认**：

| 级别 | 包含动作 | MVP | v1+ |
|---|---|---|---|
| 只读 | 浏览、搜索、抽取、总结 | ✅ 自动 | ✅ |
| 低风险写入 | 填表、草稿写入、生成导出文件 | ✅ **需显式确认** | ✅ |
| 高风险写入 | 发邮件、下单、支付、发布、删除 | ❌ | 逐步开放，强制二次确认 |

执行原则：动作前预览 → 用户确认 → 执行 → 失败自动切只读模式。

## 8. 安全、隐私与合规要求

- **数据隔离与删除**：对话/文件/项目/记忆按用户隔离；提供一键删除、导出、关闭记忆、关闭训练共享，且作为**用户可见设置**而非埋在政策文字里。
- **默认不用于训练**：底层 API 数据默认不用于训练；预留 ZDR / 数据保留策略抽象。
- **浏览器与账号安全**：首版只做本地浏览器辅助，不替用户登录/持有会话。
- **政策与合规**（Day 1 预留）：Terms、Privacy Policy、Refund Policy、Copyright/DMCA、DSAR 工单、内容安全策略、未成年人限制、分区域数据驻留抽象。

## 11. 范围边界（明确不做什么）

为避免"做全家桶"的产品过宽风险，MVP **明确不做**：
- 高自治、无确认的自动执行。
- 云浏览器登录 / 替用户持有会话。
- 团队协作、Connectors 生态、BYOK / Enterprise（均为 v2）。
- 移动原生 App（v1 后用轻壳复用 Web 能力）。
- 自训练 / 微调模型（把 GPT/Claude 当可替换能力提供者）。

## 12. 技术能力增强：借鉴 OpenClaw 与 Hermes Agent

> 来源：[openclaw/openclaw](https://github.com/openclaw/openclaw)（MIT，自托管个人 AI 助手网关）与 [NousResearch/hermes-agent](https://github.com/NousResearch/hermes-agent)（OpenClaw 谱系演进，强在闭环学习与持久记忆）。
>
> **取舍原则**：两者本质是"自托管 + 多消息渠道（WhatsApp/Telegram/…）+ 全自治"的个人 agent 网关。ApollaAI 是托管型 To C 研究→成品工作台，因此**只吸收可迁移的工程能力，刻意不引入**多消息渠道接入、自托管优先、语音唤醒、原生伴侣 App 等（这会触发报告警示的"做全家桶"风险）。

### 12.A 声明式 Skill 文件 + 闭环自动写 Skill
- **声明式格式**（借鉴 OpenClaw `SKILL.md` / `AGENTS.md` / `TOOLS.md`，兼容 [agentskills.io](https://agentskills.io) 开放标准）：每个 Skill = 一个带 frontmatter 的 Markdown（名称、触发场景、所需工具、I/O Schema、安全级别、Prompt 版本），运行时注入上下文。与 §架构 的 Prompt Registry 同源管理，便于版本化、灰度、回滚、分享。
- **闭环自动写 Skill**（借鉴 Hermes "closed learning loop"）：当一次任务被高质量收尾（用户点赞 / 导出 / 分享）时，系统**自动起草一个可复用 Skill** 供用户一键审阅保存——把 PRD §4 的"用户流三（模板化复用）"从手动升级为自动。Skill 在复用中可被增量改进（self-improve）。
- 版本：声明式格式在架构期落地；自动写 Skill 在 v1（P1）。

### 12.B 持久记忆架构（让 Memory Lite 可落地）
借鉴 Hermes 的多层记忆，给 §5 的 Memory Lite 一个具体可实现架构：
- **会话级检索**：Postgres FTS（或 FTS5 等价）全文检索历史会话 + LLM 摘要，实现跨会话回忆。
- **结构化用户模型**（USER.md 等价）：写作风格、领域偏好、项目习惯、常用格式，作为可编辑、用户可见的档案。
- **Memory nudging**：在合适时机（新任务、风格切换）按相关性注入记忆，而非每次全量塞入——与缓存/成本策略一致。
- 全部记忆受 §8 数据控制：可见、可编辑、可一键删除、可关闭。
- 版本：v1（P1）。

### 12.C MCP 作为工具与连接器标准
- 借鉴 Hermes "MCP out of the box"：**Tool Runtime 以 [Model Context Protocol](https://modelcontextprotocol.io) 作为外部工具/数据源的统一接入标准**，而不是为每个 SaaS 写 bespoke 连接器。
- 收益：复用社区 MCP 生态（呼应原报告中 Genspark "700+ MCP" 的护城河），把 §5 的 Connectors（v2）从"逐个集成"变为"接入标准"。
- 安全：每个 MCP server 的工具同样受 §7 三级权限与 §12.E 不可信输入约束。
- 版本：Tool Runtime 在架构期预留 MCP 抽象；基础 MCP 客户端 v1（面向 power user）；连接器市场 v2。

### 12.D 沙箱执行运行时（安全 + 成本）
借鉴 OpenClaw 的"每会话沙箱 + 工具 allowlist/denylist"与 Hermes 的"多后端终端 + serverless 休眠"：
- **每任务工具白名单**：默认只读；低风险写入工具需显式授予；高风险工具默认拒绝（对齐 §7）。
- **多后端隔离**：默认 Docker 容器隔离用于代码执行 / Artifact 运行 / 浏览器辅助动作；非可信内容触发的执行强制进沙箱。
- **Serverless 休眠**（Modal/Daytona 式）：异步任务 Worker 与 Artifact 运行环境**空闲休眠**，"会话间几乎零成本"——直接服务 PRD §9 的"模型/工具成本占收入比"目标。
- 版本：基础沙箱（Docker + 工具白名单）在 MVP（支撑 Artifact 与研究工具）；serverless 休眠与浏览器动作沙箱在 v1。

### 12.E 不可信输入防护（防 Prompt 注入）— P0
借鉴 OpenClaw 的 `dmPolicy` / pairing /"inbound 视为不可信"的安全心智。**这是 ApollaAI 当前文档缺失、但 MVP 必需的安全能力**：MVP 即会摄入网页、上传文件、工具输出等外部内容，全是 prompt 注入面。
- **内容隔离**：外部内容以"数据"而非"指令"通道进入模型，加固系统 Prompt 边界。
- **动作门控**：当某个写入/执行动作是由不可信内容"建议"产生时，强制走 §7 的显式确认，且默认降级为只读。
- **来源标注**：在证据面板标出哪些片段来自不可信外部源。
- 版本：MVP（P0），归入 Safety & Policy Engine。

### 12.F 上下文压缩 + 操作指令
借鉴 OpenClaw 的 chat 指令（`/compact`、`/think`、`/usage`、`/new`、`/reset`、`/trace`）：
- **上下文压缩**（`/compact` 等价）：长 Research / 长会话超阈值时自动或手动压缩，是长任务可行性与成本控制的刚需。
- **思考档位**（`/think`）：对应模型路由的推理强度档位。
- **用量指令**（`/usage`）：随时查看当前任务成本，呼应 §6.8 成本面板。
- 版本：上下文压缩在 MVP（P0，长 Research 必需）；完整 slash 指令集在 v1 打磨。

### 12.G 交互式 Artifact（Agent-to-UI / A2UI）
借鉴 OpenClaw 的 Live Canvas / A2UI：把 §6.5 的 Artifact 从"静态生成网页"升级为 **agent 可控的交互界面**——agent 可渲染组件、捕获用户输入、按输入更新产物。这是相对竞品的差异化抓手。
- 版本：MVP 保持静态 Artifact；交互式 A2UI 在 v1（P1）。

### 12.H 模型 failover 链与多密钥/订阅
借鉴 OpenClaw 的"model failover + auth profile rotation + OAuth subscriptions"，增强 §6.1 Model Router：
- **显式 failover 链**：每个逻辑别名配多个候选，按可用性/速率/成本顺序回退（比当前"高阶失败→轻量"更系统）。
- **多密钥轮换**：跨多把 API key 轮换以抗速率限制。
- **OAuth 订阅**：可选用 ChatGPT/Claude 订阅额度而非纯 API（成本优化，归 BYOK）。
- 版本：failover 链 + 多密钥轮换在架构期/阶段1（P0）；OAuth 订阅在 v2（BYOK）。

### 12.I 不引入的能力（明确边界）
为守住定位，**明确不从两者引入**：多消息渠道接入（WhatsApp/Telegram/Discord/iMessage…）、自托管优先模式、语音唤醒、原生伴侣 App、全自治无确认执行。这些与 ApollaAI 的"托管 To C 研究工作台 + 低风险执行"定位冲突。

## 13. 多媒体生成（文生图 / 文生视频）

> 目标：让 ApollaAI 的"成品力"从文档/Slides/网页扩展到**图像与视频**，服务内容创作者/营销画像；并以**统一适配层**灵活对接多家图像/视频模型（如 Seedance 2.0），与 §6.1 的 LLM 路由同构——**配置即换模型，不在业务里硬编码模型名**。

### 13.1 设计原则
- **与 LLM 路由同构**：媒体模型同样用"逻辑别名 + provider 适配器 + 路由配置中心"。
- **异步任务对象**：图像（秒级）可半同步，视频（十秒~分钟级）一律走 §架构 的异步任务对象，前端先显示任务计划/进度/预估耗时与预估成本。
- **成品化**：产物自动落入 §6.5 Artifact / §6.6 导出与素材库，可预览、编辑参数重生成、分享、导出。
- **成本可见**：媒体生成单价高，必须任务级成本预估 + 实际成本写入 §6.8 UsageLedger，高成本任务显式二次确认。
- **内容安全**：生成前 Prompt 审核 + 生成后内容审核（NSFW/版权/名人肖像等），纳入 §8 与 Safety & Policy Engine。

### 13.2 媒体模型适配层（核心：灵活对接多模型）
统一 `MediaAdapter` 接口，新模型只需实现适配器并在路由配置登记即可上线，无需改业务代码：

| 接口能力 | 说明 |
|---|---|
| `submit(job)` | 提交生成任务（文生图/图生图/文生视频/图生视频/首尾帧/参考图）|
| `poll(jobId)` / 回调 | 轮询或 webhook 获取进度与结果 |
| `fetchResult(jobId)` | 拉取产物（图/视频 URL 或二进制），转存对象存储 |
| `capabilities()` | 能力矩阵：分辨率、时长、宽高比、帧率、是否支持参考图/运镜/音轨、最大并发 |
| `estimateCost(params)` | 按参数预估成本/积分 |
| `mapError()` | 统一错误码与失败降级 |

**逻辑别名（示例，实际映射在路由配置中心）**：

| 别名 | 默认职责 | 候选 provider（可配置）|
|---|---|---|
| `image_fast` | 快速出图、缩略、草图 | 低成本图像模型 |
| `image_premium` | 高质量出图、图生图、精修 | 高阶图像模型（如 Seedream / Imagen / Flux / GPT-image 等）|
| `video_standard` | 标准短视频（数秒）| 中档视频模型 |
| `video_premium` | 高质量/长时/强一致性视频 | **Seedance 2.0** / Kling / Veo / Runway 等 |

> Provider 名称仅为占位，实际以路由配置与内部评测择优；接口与 BYOK（§5）兼容，可由用户自带密钥调用其授权的模型。

### 13.3 Seedance 2.0 视频模型对接（首批 video provider）
- 以 `SeedanceVideoAdapter` 实现统一 `MediaAdapter`：支持文生视频/图生视频/参考图与运镜参数、宽高比、时长、分辨率、帧率，按其异步任务/回调机制对接。
- 凭证与端点走环境变量/密钥管理（§12.D），密钥加密存储；区域可用性纳入 §16 开放问题（首发市场决定可达性与合规）。
- 能力矩阵与计费参数在 `capabilities()`/`estimateCost()` 中声明，供路由与成本预估使用。
- 验收：能在统一工作区"输入提示词/参考图 → 生成 → 预览 → 重生成/导出/分享"完成闭环。

### 13.4 用户流（融入既有三流）
- **研究到成品**（§4 用户流一）追加产物类型：报告/Slides/网页 之外可"一键配图/生成封面/生成讲解短视频"。
- **媒体创作独立入口**：主工作台输入框支持"图像""视频"模式（与 Auto/GPT/Claude 模式并列），选别名、设参数、看预估成本后提交。
- 媒体产物可保存为 §12.A 的 Skill 模板（如"小红书封面风格""产品讲解短视频模板"）复跑。

### 13.5 指标补充
- 媒体生成成功率（产物可用且无重试退出）≥ 70%（MVP n/a，v1 目标）。
- 媒体任务成品转化率（生成后导出/分享/采用）≥ 35%。
- 媒体成本占媒体相关收入比 ≤ 40%（受高阶视频模型影响，单列监控）。

### 13.6 版本与边界
- **v1（P1）**：文生图 + 媒体适配层 + 至少 1 个图像 provider；文生视频 + Seedance 2.0 适配（视频可视容量/成本压力定 P1 末或 P2 初）。
- **v2（P2）**：更多 provider、批量/队列、风格库、品牌一致性、视频时间线编辑。
- **不做（首期）**：长视频非线性剪辑器、专业调色/音轨工作站（超出工作台定位，避免做全家桶）。

| **子代理 / 编排（并行分工）** | Orchestrator 可派生子代理并行执行、汇总 | 🆕 新增（显式化）|
| **不确定时主动澄清** | 关键决策/高风险动作前主动提问，而非擅自行动 | 🆕 新增（行为规范）|
| **在你的文件里工作（桌面文件工作区）** | 桌面端授权目录读写，作为任务工作区 | 🆕 新增（桌面）|
| Slash 命令 | /compact、/think、/usage 等 + Plugin 自带命令 | 复用 §12.F |
| 持久记忆 | 跨会话记忆与用户模型 | 复用 §12.B |
| 桌面 App（macOS/Windows）| 桌面端从轻壳升级为可授权文件/应用的 Cowork 宿主 | 扩展 §架构 Desktop |
| 企业管控（RBAC、组织花费上限、用量分析、OpenTelemetry、按工具连接器开关）| 见 §15.4 | 🆕 扩展 §14.3 |

### 15.2 Plugins（按职能打包，对标 Cowork 核心）
- 一个 Plugin = **Skills + 连接器 + slash 命令 + 子代理** 的角色化捆绑（如"销售""市场研究""个人助理""开发者"）。
- 提供官方 Plugin（一键安装）与自建 Plugin；Plugin 声明其所需连接器与权限，安装时统一授权。
- 与 §12.A 声明式 Skill 同源管理，可版本化、分享、上架模板/Skill 市场（§5 Share & Publish）。

### 15.3 安全与信任（Cowork 模式默认更严）
- **沙箱/VM 隔离**：所有执行在隔离环境，受 §12.D 工具 allowlist 约束。
- **权限分级**（§7）：只读自动；低风险写入需确认；高风险（发邮件/支付/发布/删除）强制二次确认——Cowork **不绕过**该机制。
- **主动澄清**：信息不足或动作不可逆时停下来问，而不是猜测执行。
- **不可信输入防护**（§12.E）：连接器/网页/文件返回内容走数据通道，防 prompt 注入越权。
- **按工具连接器开关**：用户/管理员可逐个工具开关与限权。

### 15.4 企业能力（对齐 Cowork GA 企业特性，v2）
RBAC 角色权限、组织/组花费上限、扩展用量分析、OpenTelemetry 观测接入、按工具的连接器控制；与 §14.3 团队治理合并交付。

### 15.5 版本与边界
- **底座先行**：沙箱（MVP）、MCP 连接器（v1/v2）、Skills（v1）、记忆（v1）、后台/定时任务（v1）按既有节奏交付。
- **Cowork 模式整合**：在底座成熟后于 **v2** 作为旗舰模式上线（Plugins + 子代理 + 桌面文件工作区 + 企业管控）。
- **守护栏**：Cowork 不改变"低风险自动化 + 高风险确认"的根本原则；不做无确认的全自治、不做替用户持有敏感账号会话的危险授权。
